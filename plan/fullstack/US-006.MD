# Plan: US-006 — AI Chat Interface (Backend + Frontend)

## Context

US-001 through US-005 and US-007 are fully implemented. The app has 18 REST endpoints across team members, schedule requirements, and schedule entries. US-006 adds an AI-powered floating chat widget that lets the admin control all scheduling features through natural language. The AI proposes tool calls, the user reviews each in a structured visual card and explicitly approves or rejects before execution.

**Key PRD reference**: `plan/PRD.md` lines 182–225 (AC-006-A through AC-006-R)
**AI SDK reference**: `plan/creating-an-ai-chat.md` — Vercel AI SDK tool approval pattern

---

## Technology Choice: Vercel AI SDK + Gemini

Use `ai` (Vercel AI SDK) + `@ai-sdk/google` with a Gemini model.

**Why**: The reference doc demonstrates Vercel AI SDK patterns (`streamText`, `tool({ needsApproval: true })`, `useChat`, `addToolApprovalResponse`). This provides the entire tool approval flow (AC-006-D through AC-006-J) out of the box. The Vercel AI SDK is provider-agnostic — swapping the provider package (`@ai-sdk/google` instead of `@ai-sdk/anthropic`) is the only change needed. Gemini models have strong tool-calling capabilities suitable for this use case.

## System Prompt Design (`system-prompt.ts`)

`buildSystemPrompt()` returns a string with:
- Role: "Hospital scheduling assistant"
- Today's date (dynamically injected)
- Professions + valid specialties (exact enum values from `team-member.ts`)
- Date reference explanation (weekday, weekend, day names, "1st of the month")
- Tool usage guidelines:
  - Always `listTeamMembers` before modifying (to resolve names → IDs)
  - Always `listScheduleRequirements` before modifying requirements
  - Always `listScheduleEntries`/`getScheduleOverview` before swaps
  - `weekStartDate` must be a Monday for auto-fill
  - Map natural language to exact enum values ("neurologist" → DOCTOR + NEUROLOGY)
- Ask for clarification when ambiguous (AC-006-O)

## Chat Route Handler (`routes/chat.ts`)

```ts
export async function chat(req: Request, res: Response) {
  const { messages } = chatSchema.parse(req.body)
  const result = streamText({
    model: createChatModel(),
    system: buildSystemPrompt(),
    messages,
    tools: allTools,
    maxSteps: 10,
  })
  result.pipeDataStreamToResponse(res)
}
```

Registered in `app.ts` as `app.post('/chat', chat)` before team member routes.

## Model Configuration (`model.ts`)

```ts
export function createChatModel() {
  const google = createGoogleGenerativeAI({ apiKey: env.GOOGLE_GENERATIVE_AI_API_KEY })
  return google('gemini-2.5-flash')
}
```

Uses Gemini 2.5 Flash for fast, capable tool calling with good cost efficiency. Model ID is isolated here for easy swapping.

## Tool Error Handling Pattern

Each tool's `execute` wraps the use case call:
```ts
execute: async (params) => {
  try {
    // ... use case execution + presenter
    return { success: true, data: presentedResult }
  } catch (error) {
    if (error instanceof DomainError) {
      return { success: false, error: error.response.code }
    }
    if (error instanceof ValidationError) {
      return { success: false, error: error.response.code, field: error.response.errorField }
    }
    return { success: false, error: 'UNEXPECTED_ERROR' }
  }
}
```

Returns error info as tool result (not thrown) so the AI gets context to provide a helpful message.

## Chat State Management (Jotai Atoms)

**File: `layout/states/chat-atoms.ts`**

Three atoms for cross-page state:
- `chatOpenAtom` — widget visibility toggle
- `chatInputAtom` — pending input from contextual entry points
- `chatInputFocusTriggerAtom` — counter to trigger textarea focus

Uses Jotai's default store (no Provider needed), consistent with existing `useDialogsAtom` pattern.

## Chat Widget Component Details

### `ChatWidget.tsx` — Root orchestrator

Renders in `AppLayout` (outside Route outlets → persists across navigations, AC-006-B).

Owns the `useChat` hook:
```ts
const chatHook = useChat({
  api: 'http://localhost:5000/chat',
})
```

Watches `chatInputAtom` via `useEffect` — when non-empty, calls `chatHook.setInput(pendingInput)` and resets atom.

Renders: `ChatToggleButton` (always) + `ChatPanel` (when `chatOpenAtom` is true).

### `ChatToggleButton.tsx` — Floating action button

`fixed bottom-6 right-6 z-[var(--z-chat)]`. Circular 48x48 button, `bg-primary-500`, `MessageCircle` icon → `X` icon when open. `scale-in` animation.

### `ChatPanel.tsx` — Chat panel

`fixed bottom-20 right-6 z-[var(--z-chat)]`. Width: `w-[420px]`, height: `h-[600px] max-h-[80vh]`. `rounded-2xl`, `shadow-xl`, `bg-surface`, `border border-border`. `slide-in-right` animation.

Header: "AI Assistant" + `Bot` icon + close button.
Body: `ChatMessageList` (flex-1, overflow-y-auto).
Footer: `ChatInput`.

### `ChatMessageList.tsx` — Message feed

Maps `messages` from `useChat`. For each message:
- `role: 'user'` → right-aligned bubble (`ChatMessage`)
- `role: 'assistant'` → left-aligned, with text parts rendered as `ChatMessage` and tool invocation parts rendered as `ToolApprovalMessage`

Auto-scrolls to bottom on new messages via `useRef` + `useEffect`.

### `ChatMessage.tsx` — Single message bubble

User: `bg-primary-50 text-text-primary`, right-aligned.
Assistant: `bg-surface-sunken`, left-aligned.
Text content with `text-body-sm` styling.

### `ChatInput.tsx` — Input area

Auto-growing `textarea` (max 4 rows) + Send button (`Button variant="primary" size="sm"`, `Send` icon).
Binds to `useChat`'s `input`, `handleInputChange`, `handleSubmit`.
Enter (without Shift) submits.
Watches `chatInputFocusTriggerAtom` for focus trigger.

### `ToolApprovalMessage.tsx` — Collapsed tool call

Renders for tool invocation parts in assistant messages.
Shows tool display name + brief param summary + "Review Action" button.
Local `isExpanded` state — when expanded, renders `ToolApprovalCard`.
When tool state is `'result'`, shows result summary instead.

### `ToolApprovalCard.tsx` — Expanded tool details

Maps tool name → specific card component via `TOOL_CARD_MAP` record (falls back to `GenericToolCard`).
Approve: `addToolApprovalResponse({ toolCallId, approved: true })` → after result, calls `invalidateToolQueries`.
Reject: `addToolApprovalResponse({ toolCallId, approved: false })` + `append({ role: 'system', content: ... })`.

## Tool-Specific Visual Cards (AC-006-F)
