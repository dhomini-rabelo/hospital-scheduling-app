# Plan: US-002 Backend — Define Schedule Requirements

## Context

US-001 (Team Members) is complete. US-002 requires backend CRUD for schedule requirements: rules that define staffing needs per date reference. A single schedule requirement for a date reference (e.g., "weekday") contains **all profession counts and their specialty breakdowns** stored as a JSON field.

**Key PRD requirements (backend-relevant):**
- AC-002-A: Create with profession(s), date reference (free-form string), required counts
- AC-002-B: Specialty sub-requirements per profession (e.g., "of 5 Doctors, 2 must be Neurologists")
- AC-002-C: No duplicates for same date reference (one rule per date reference)
- AC-002-D: Sum of specialty counts <= profession's total required count
- AC-002-E: Edit required counts, specialty sub-requirements, isEnabled
- AC-002-F: Delete a schedule requirement
- AC-002-G: Enable/disable (disabled = ignored by auto-fill but persisted)
- AC-002-I: Persist in SQLite

---

## Design Decisions

1. **One schedule requirement per date reference** — contains all professions and their specialty breakdowns in a single JSON field
2. **`dateReference` is a plain string** — PRD says "user-defined labels" (e.g., "weekday", "1st of the month")
3. **JSON field pattern** — using `ValueObject` > `JsonField` > `RequirementsDataJsonField` per `plan/json-field-guide.md`
4. **Shared validation** in `domain/use-cases/utils/schedule-requirement.ts` — used by both create and update use cases
5. **Enable/disable is part of PUT** — follows existing update pattern
6. **Reuse `PROFESSION_SPECIALTIES`** from `team-member.ts` for specialty validation

---

## JSON Field Data Shape

```ts
type SpecialtyRequirement = {
  specialty: string       // e.g., "NEUROLOGY" — validated against PROFESSION_SPECIALTIES
  requiredCount: number   // positive integer
}

type ProfessionRequirement = {
  profession: string              // e.g., "DOCTOR" — validated against Profession enum
  requiredCount: number           // total for this profession
  specialtyRequirements: SpecialtyRequirement[]  // optional breakdown
}

// The JSON field holds an array of ProfessionRequirement
type RequirementsData = ProfessionRequirement[]
```

Example value:
```json
[
  { "profession": "DOCTOR", "requiredCount": 5, "specialtyRequirements": [
    { "specialty": "NEUROLOGY", "requiredCount": 2 },
    { "specialty": "CARDIOLOGY", "requiredCount": 1 }
  ]},
  { "profession": "NURSE", "requiredCount": 3, "specialtyRequirements": [
    { "specialty": "ICU", "requiredCount": 1 }
  ]}
]
```

## Prisma Schema (1 model, no relations)

```prisma
model ScheduleRequirement {
  id            String   @id @default(uuid())
  dateReference String   @unique @map("date_reference")
  requirements  String   // JSON string of RequirementsData
  isEnabled     Boolean  @default(true) @map("is_enabled")
  createdAt     DateTime @default(now()) @map("created_at")
  updatedAt     DateTime @updatedAt @map("updated_at")

  @@map("schedule_requirements")
}
```

- `@@unique` on `dateReference` enforces one rule per date reference (AC-002-C)
- `requirements` is a String storing the JSON (via `JsonField.toValue()`)
- No separate table for specialty sub-requirements

## Files to Create (15 new)

### Layer 0: Foundation — ValueObject + JsonField
These are generic building blocks used by the JSON field pattern.

**`backend/src/modules/domain/value-object.ts`**
```ts
export abstract class ValueObject<Value = any, Response = any> {
  public readonly value: Readonly<Value>
  protected constructor(value: Value) { this.value = value }
  abstract toValue(): Response
}
```

**`backend/src/modules/domain/json-field.ts`**
```ts
export class JsonField<Value = any> extends ValueObject<Value, string> {
  public toValue(): string { return JSON.stringify(this.value) }
}
```

### Layer 1: Domain Entity

**`backend/src/domain/entities/schedule-requirement.ts`**
- Types: `SpecialtyRequirement`, `ProfessionRequirement`, `RequirementsData`
- `RequirementsDataJsonField extends JsonField<RequirementsData>` with `static create(value)`
- `ScheduleRequirementProps`: dateReference (string), requirements (RequirementsDataJsonField), isEnabled (boolean), createdAt, updatedAt
- `ScheduleRequirement extends Entity<ScheduleRequirementProps>` with `create()` and `reference()`
- `create()` defaults `isEnabled: true` and sets timestamps

### Layer 2: Shared Validation (use-cases/utils)

**`backend/src/domain/use-cases/utils/schedule-requirement.ts`**
- `validateRequirementsData(requirements: RequirementsData)`:
  - Each `profession` must be a valid `Profession` enum value
  - No duplicate professions in the array
  - Each `requiredCount` must be a positive integer
  - Each `specialty` must be valid for its parent profession (reuse `PROFESSION_SPECIALTIES`)
  - No duplicate specialties within a profession
  - Sum of specialty `requiredCount` values <= profession `requiredCount` (AC-002-D)
  - At least one profession requirement must be provided
- Throws `ValidationError` with descriptive field paths and codes

### Layer 3: Abstract Repository

**`backend/src/adapters/repositories/schedule-requirement-repository.ts`**
```ts
abstract class ScheduleRequirementRepository {
  abstract create(input: CreateInput): Promise<ScheduleRequirement>
  abstract findById(id: string): Promise<ScheduleRequirement | null>
  abstract findByDateReference(dateReference: string): Promise<ScheduleRequirement | null>
  abstract findAll(): Promise<ScheduleRequirement[]>
  abstract update(id: string, input: UpdateInput): Promise<ScheduleRequirement>
  abstract delete(id: string): Promise<void>
}
```
- `CreateInput`: `{ dateReference, requirements: RequirementsDataJsonField, isEnabled? }`
- `UpdateInput`: `{ requirements?: RequirementsDataJsonField, isEnabled?: boolean }`
- `findByDateReference` used by create use case to enforce uniqueness (AC-002-C)

### Layer 4: Prisma Repository

**`backend/src/infra/adapters/repository/prisma-schedule-requirement-repository.ts`**
- `toEntity()` mapper: parses `requirements` string via `RequirementsDataJsonField.create(JSON.parse(record.requirements))`
- `create()`: stores `requirements` via `input.requirements.toValue()` (JSON.stringify)
- `update()`: conditionally includes `requirements: input.requirements.toValue()` if provided
- `findByDateReference()`: uses `findFirst({ where: { dateReference } })`
- `findAll()`: orders by `createdAt: 'desc'`

### Layer 5: Use Cases (4 files)

**`backend/src/domain/use-cases/schedule-requirement/create-schedule-requirement.ts`**
- Payload: `{ dateReference, requirements: RequirementsData, isEnabled? }`
- Calls `validateRequirementsData(payload.requirements)`
- Calls `repository.findByDateReference(payload.dateReference)` — if found, throw `DomainError(DATA_INTEGRITY, 'DUPLICATE_DATE_REFERENCE')`
- Creates `RequirementsDataJsonField.create(payload.requirements)` and calls `repository.create()`

**`backend/src/domain/use-cases/schedule-requirement/list-schedule-requirements.ts`**
- No payload, returns `{ items: ScheduleRequirement[] }`

**`backend/src/domain/use-cases/schedule-requirement/update-schedule-requirement.ts`**
- Payload: `{ id, requirements?: RequirementsData, isEnabled?: boolean }`
- Finds by ID (404 if missing)
- If `requirements` is provided, calls `validateRequirementsData(payload.requirements)`
- Builds update input, calls `repository.update()`

**`backend/src/domain/use-cases/schedule-requirement/delete-schedule-requirement.ts`**
- Payload: `{ id }`
- Finds by ID (404 if missing), calls `repository.delete()`

### Layer 6: Presenter

presentScheduleRequirement(item: ScheduleRequirement) -> {

## API Contract

| Method | Path | Status | Response |
|--------|------|--------|----------|
