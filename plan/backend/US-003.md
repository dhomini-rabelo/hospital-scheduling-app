# Plan: US-003 Backend — View & Create Schedules

## Context

US-001 (Team Members) and US-002 (Schedule Requirements) are complete. US-003 introduces **ScheduleEntry** — one per day, holding a `structure` JSON field (the staffing plan for that day) and a many-to-many relationship to team members (the assigned staff). The backend must support: setting (create/update) entries from today through next week, viewing a daily overview with requirement fulfillment/gap analysis, listing entries, and deleting entries.

---

## Design Decisions

### 1. One ScheduleEntry per day with a `structure` JSON field

Each ScheduleEntry has a unique `date` (String, YYYY-MM-DD) and a `structure` field (JSON, same shape as `RequirementsData` from ScheduleRequirement). The structure defines the staffing plan for that day — how many of each profession/specialty are needed.

### 2. Many-to-many relation to TeamMember via WatchedList pattern

Following `plan/many-to-many-relationship.md`:
- **Map entity**: `ScheduleEntryTeamMemberMap` (junction entity with scheduleEntryId + teamMemberId)
- **WatchedList**: `ScheduleEntryTeamMemberMapWatchedList` (tracks additions/removals for efficient persistence)
- **Entity prop**: `ScheduleEntry.props.teamMembers` holds the WatchedList

### 2b. Many-to-many relation to ScheduleRequirement via WatchedList pattern

Tracks which schedule requirements were selected when creating/updating an entry:
- **Map entity**: `ScheduleEntryScheduleRequirementMap` (junction entity with scheduleEntryId + scheduleRequirementId)
- **WatchedList**: `ScheduleEntryScheduleRequirementMapWatchedList` (tracks additions/removals for efficient persistence)
- **Entity prop**: `ScheduleEntry.props.scheduleRequirements` holds the WatchedList

### 3. `set-schedule-entries` use case — upsert with flat payload

The payload is flat: `{ dates, structure, teamMemberIds, scheduleRequirementIds }`. The same structure, team members, and schedule requirements are applied to ALL selected dates. This matches the typical workflow where the admin defines one staffing plan and applies it across multiple days (e.g., all weekdays).

For each date:
- If entry exists → update structure + use `WatchedList.update()` for team members
- If no entry exists → create new entry

Validates team member attachments against the structure (once, since they're shared):
- Each assigned team member's profession must appear in the structure
- Count of assigned members per profession must not exceed the structure's `requiredCount`
- Count of assigned members per specialty must not exceed the specialty's `requiredCount`

### 4. Date as String (YYYY-MM-DD)

Avoids timezone ambiguity. `@unique` on `date` ensures one entry per day.

### 5. Repository with aggregate data types

Following `plan/repositories-interface-example.md`, the repository defines `ScheduleEntryWithAggregateData` for queries that need resolved team member entities.

### 6. Overview uses both entry structure AND global requirements

- Entry's structure → shows if the day's staffing plan is met
- Global enabled schedule requirements (matching date references) → shows organizational requirement fulfillment (AC-003-F)

### 7. Date range for mutations: today → next week Sunday

The `set-schedule-entries` use case only accepts dates from today through next week's Sunday. The overview and list endpoints serve any date in current + next week range.

## Prisma Schema Changes

```prisma
model TeamMember {
  // ... existing fields unchanged ...
  scheduleEntryTeamMemberMaps ScheduleEntryTeamMemberMap[]
}

model ScheduleEntry {
  id          String   @id @default(uuid())
  date        String   @unique
  structure   String                              // JSON of StructureData (same shape as RequirementsData)
  createdAt   DateTime @default(now()) @map("created_at")
  updatedAt   DateTime @updatedAt @map("updated_at")

  teamMembers          ScheduleEntryTeamMemberMap[]
  scheduleRequirements ScheduleEntryScheduleRequirementMap[]

  @@map("schedule_entries")
}

model ScheduleEntryTeamMemberMap {
  id              String        @id @default(uuid())
  scheduleEntryId String        @map("schedule_entry_id")
  teamMemberId    String        @map("team_member_id")
  scheduleEntry   ScheduleEntry @relation(fields: [scheduleEntryId], references: [id], onDelete: Cascade)
  teamMember      TeamMember    @relation(fields: [teamMemberId], references: [id])
  createdAt       DateTime      @default(now()) @map("created_at")
  updatedAt       DateTime      @updatedAt @map("updated_at")

  @@unique([scheduleEntryId, teamMemberId])
  @@map("schedule_entry_team_members")
}
```

### Overview Response

```ts
{
  date: string
  totalAssigned: number
  entries: {
    id: string
    teamMember: { id, name, profession, specialty, createdAt, updatedAt }
  }[]
  scheduleRequirements: {                // schedule requirements attached to this entry
    id: string
    dateReference: string
    requirements: ProfessionRequirement[]
    isEnabled: boolean
    createdAt: string
    updatedAt: string
  }[]
  structureFulfillment: {              // from the entry's structure field
    professions: {
      profession: Profession
      requiredCount: number
      assignedCount: number
      isFulfilled: boolean
      specialties: {
        specialty: Specialty
        requiredCount: number
        assignedCount: number
        isFulfilled: boolean
      }[]
    }[]
  } | null                             // null if no entry exists for that date
  requirementsFulfillment: {           // from global enabled schedule requirements
    requirementId: string
    dateReference: string
    isFulfilled: boolean
    professions: {
      profession: Profession
      requiredCount: number
      assignedCount: number
      isFulfilled: boolean
      specialties: { specialty, requiredCount, assignedCount, isFulfilled }[]
    }[]
  }[]
}
```

### Layer 2: Domain Utilities

**`backend/src/domain/utils/schedule.ts`**
- `getDateReferencesForDate(dateStr: string): string[]` — maps a date to matching date reference labels (lowercase): day name ("monday"–"sunday"), category ("weekday"/"weekend"), ordinal ("1st of the month")

**`backend/src/domain/use-cases/utils/schedule-entry.ts`**

- `validateTeamMembersAgainstStructure(teamMembers: TeamMember[], structure: StructureData): void` — validates:
  - Each team member's profession must appear in the structure
  - Count per profession ≤ structure's requiredCount for that profession
  - Count per specialty ≤ structure's specialty requiredCount

### Layer 4: Prisma Repository

**`backend/src/infra/adapters/repository/prisma-schedule-entry-repository.ts`**
- `toEntity()` mapper: converts Prisma record (with included teamMembers) → builds `ScheduleEntryTeamMemberMap.reference()` for each junction record → wraps in `ScheduleEntryTeamMemberMapWatchedList` → builds `ScheduleEntry.reference()`
- `create()`: creates entry + nested creates for junction records (`teamMembers.getItems()`)
- `update()`:
  - DELETE removed junction records (`teamMembers.getRemovedItems()` → IDs)
  - INSERT new junction records (`teamMembers.getNewItems()` → scheduleEntryId + teamMemberId)
  - UPDATE entry's structure field
- `getWithAggregateData()`: includes `teamMembers.teamMember` relation → resolves full TeamMember entities
- `listWithAggregateData()`: same but for date range, ordered by date asc

### Layer 5: Use Cases (4 files)

**`backend/src/domain/use-cases/schedule-entry/set-schedule-entries.ts`**
- Payload: `{ dates: string[], structure: StructureData, teamMemberIds: string[], scheduleRequirementIds: string[] }`
- Constructor takes `ScheduleEntryRepository` + `TeamMemberRepository` + `ScheduleRequirementRepository`
- Steps:
  1. Validate all dates in allowed set range (today → next Sunday)
  2. `validateStructureData(structure)` — validate the shared structure once
  3. Fetch all team members via `teamMemberRepository.findAll()`, filter to the provided IDs, validate all exist
  4. `validateTeamMembersAgainstStructure(resolvedMembers, structure)` — validate once (same team members + structure for all dates)
  5. Fetch all schedule requirements via `scheduleRequirementRepository.findAll()`, validate all provided IDs exist
  6. For each date in payload:
     a. Check if entry exists for date (`repository.findByDate`)
     b. If exists: build map entities (team members + requirements) with existing entry's ID → `watchedList.update()` → `repository.update()`
     c. If not exists: build map entities → create WatchedLists → `repository.create()`
  7. Return `{ items: ScheduleEntry[] }`

**`backend/src/domain/use-cases/schedule-entry/list-schedule-entries.ts`**
- Payload: `{ startDate: string, endDate: string }`
- Calls `repository.listWithAggregateData(startDate, endDate)`
- Returns `{ items: ScheduleEntryWithAggregateData[] }`

**`backend/src/domain/use-cases/schedule-entry/get-schedule-overview.ts`**
- Payload: `{ date: string }`
- Constructor takes `ScheduleEntryRepository`, `TeamMemberRepository`, `ScheduleRequirementRepository`
- Steps:
  1. Fetch entry with aggregate data via `repository.getWithAggregateData(date)`
  2. Fetch all enabled schedule requirements via `scheduleRequirementRepository.findAll()`, filter `isEnabled`
  3. Compute `getDateReferencesForDate(date)`, filter requirements by matching dateReference (case-insensitive)
  4. Build `structureFulfillment` from entry's structure vs assigned team members (null if no entry)
  5. Build `requirementsFulfillment` from matching global requirements vs assigned team members
  6. Return overview

**`backend/src/domain/use-cases/schedule-entry/delete-schedule-entry.ts`**
- Payload: `{ id: string }`
- Finds by ID (404 if missing), deletes (cascade removes junction records)
