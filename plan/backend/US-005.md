# Plan: US-005 Backend — Swap Staff and Auto-fill Gaps

## Context

US-001 (Team Members), US-002 (Schedule Requirements), US-003 (Schedule Entries), and US-004 (Auto-fill Schedule) are complete. US-005 enables targeted schedule adjustments: removing a team member from a specific day, manually picking a replacement from an eligible list, or automatically filling gaps for a single day.

**PRD requirements (AC-005-A through AC-005-D)**:
- AC-005-A: Allow removing a specific team member from a specific day in the schedule
- AC-005-B: After removal, automatically assign a replacement from eligible team members (matching profession and specialty)
- AC-005-C: Allow the user to manually pick a replacement from a list of eligible team members instead of auto-assigning
- AC-005-D: Update the persisted schedule after the swap

---

## Design Decisions

### 1. Three use cases, three APIs

Split into three distinct use cases to match the user's workflow:

1. **List Swap Candidates** (AC-005-C) — get eligible replacement team members for a given team member on a given entry, with weekly assignment counts for fair decision-making
2. **Swap Team Member** (AC-005-A + AC-005-C + AC-005-D) — remove one team member and add another on a specific entry
3. **Auto-fill Day Gaps** (AC-005-B + AC-005-D) — automatically fill all unfilled slots in a single day's entry using fair distribution

### 2. Specialty slot detection for candidate filtering

When listing swap candidates, determine whether the removed team member was filling a specialty slot or a general profession slot:
- Check if the entry's structure has a specialty requirement matching the removed member's specialty
- Count how many currently assigned members (excluding the one being removed) have that specialty
- If `assignedOfThatSpecialty < specialtyRequiredCount` → removing this person creates a specialty gap → filter candidates by **profession AND specialty**
- Otherwise → filter candidates by **profession only**

This ensures the candidate list is relevant to the gap that would be created.

### 3. Fair distribution uses the same week scope

Weekly assignment counts for swap candidates and auto-fill gaps are computed from the same week as the entry's date (Monday through Sunday). This is consistent with US-004's approach.

### 4. Auto-fill day gaps reuses the auto-fill algorithm pattern

The single-day auto-fill follows the same two-phase approach as `AutoFillScheduleUseCase`:
- Phase 1: Fill specialty slots (most constrained first)
- Phase 2: Fill remaining general profession slots

The key difference: it preserves existing assignments and only fills unfilled slots, rather than replacing all assignments.

### 5. Export `getMondayOfWeek` and `getSundayOfWeek` from schedule utils

These are currently private in `domain/utils/schedule.ts`. Export them so the new use cases can compute the week range from an entry's date.

## API Contract

| Method | Path | Body / Params | Success | Errors |
|--------|------|--------------|---------|--------|
| GET | `/schedule-entries/:entryId/swap-candidates/:teamMemberId` | Params: entryId (uuid), teamMemberId (uuid) | 200 | 404 (entry/member not found), 400 (member not assigned to entry) |
| POST | `/schedule-entries/:entryId/swap` | Body: `{ removeTeamMemberId, addTeamMemberId }` | 200 | 404 (entry/members not found), 400 (validation) |
| POST | `/schedule-entries/:entryId/auto-fill-gaps` | Params: entryId (uuid) | 200 | 404 (entry not found) |

## Response Shapes

### List Swap Candidates Response

```ts
{
  swapContext: {
    profession: Profession
    specialty: Specialty | null       // null if member fills a general slot, not a specialty slot
  }
  candidates: {
    teamMember: { id, name, profession, specialty, createdAt, updatedAt }
    weekAssignmentCount: number       // assignments in the same week for fair display
  }[]                                 // sorted by weekAssignmentCount ascending, then name
}
```

### Swap Team Member Response

```ts
{
  entry: { id, date, structure, createdAt, updatedAt }
}
```

### Auto-fill Day Gaps Response

```ts
{
  entry: { id, date, structure, createdAt, updatedAt }
  gapReport: {
    hasGaps: boolean
    professionGaps: {
      profession: Profession
      requiredCount: number
      assignedCount: number
      deficit: number
      specialtyGaps: {
        specialty: Specialty
        requiredCount: number
        assignedCount: number
        deficit: number
      }[]
    }[]
  }
}
```

## Files to Create (6 new)

### Layer 1: Use Cases (3 files)

**`backend/src/domain/use-cases/schedule-entry/list-swap-candidates.ts`**

- Constructor takes `ScheduleEntryRepository`, `TeamMemberRepository`
- Payload: `{ entryId: string, teamMemberId: string }`
- Response: `{ swapContext, candidates }`
- Steps:
  1. Find entry by ID via `scheduleEntryRepository.findById()` → 404 if not found
  2. Verify `teamMemberId` is in entry's team member maps → 400 `TEAM_MEMBER_NOT_ASSIGNED_TO_ENTRY` if not found
  3. Find the team member via `teamMemberRepository.findById()` → 404 if not found
  4. Determine if the member fills a specialty slot:
     - Find the profession requirement in entry's structure matching the member's profession
     - Find the specialty requirement matching the member's specialty (if any)
     - Count currently assigned members with the same specialty (from entry's team member maps, resolved via `teamMemberRepository.findAll()` filtered by profession)
     - If `specialtyReq` exists AND `(assignedWithSpecialty - 1) < specialtyReq.requiredCount` → `isSpecialtySlot = true`
  5. Get all team members of the matching profession via `teamMemberRepository.findAll(profession)`
  6. Filter: exclude those already assigned to this entry (from entry's team member maps)
  7. If `isSpecialtySlot`: also filter by the member's specialty
  8. Compute weekly assignment counts:
     - Compute the week range from entry's date using `getMondayOfWeek()` / `getSundayOfWeek()`
     - Load all entries for the week via `scheduleEntryRepository.findByDateRange()`
     - Count each candidate's appearances across the week's entries' team member maps
  9. Return candidates sorted by `weekAssignmentCount` ascending, then `name` ascending
  10. Return `swapContext`: `{ profession, specialty: isSpecialtySlot ? member.specialty : null }`

**`backend/src/domain/use-cases/schedule-entry/swap-team-member.ts`**

- Constructor takes `ScheduleEntryRepository`, `TeamMemberRepository`
- Payload: `{ entryId: string, removeTeamMemberId: string, addTeamMemberId: string }`
- Response: `{ item: ScheduleEntry }`
- Steps:
  1. Find entry by ID → 404 if not found
  2. Verify `removeTeamMemberId` is in entry's team member maps → 400 `TEAM_MEMBER_NOT_ASSIGNED_TO_ENTRY`
  3. Verify `addTeamMemberId` is NOT already in entry's team member maps → 400 `TEAM_MEMBER_ALREADY_ASSIGNED_TO_ENTRY`
  4. Find remove team member via `teamMemberRepository.findById()` → 404 `TEAM_MEMBER_NOT_FOUND`
  5. Find add team member via `teamMemberRepository.findById()` → 404 `TEAM_MEMBER_NOT_FOUND`
  6. Validate profession match: `addTeamMember.profession === removeTeamMember.profession` → 400 `PROFESSION_MISMATCH`
  7. Determine if the removed member fills a specialty slot (same logic as list-swap-candidates):
     - If specialty slot: validate `addTeamMember.specialty === removeTeamMember.specialty` → 400 `SPECIALTY_MISMATCH`
  8. Remove the old map entity from watched list via `entry.props.teamMembers.remove(oldMapEntity)`
  9. Add new map entity via `entry.props.teamMembers.add(ScheduleEntryTeamMemberMap.create({ scheduleEntryId: entry.id, teamMemberId: addTeamMemberId }))`
  10. Persist via `scheduleEntryRepository.update(entry.id, { structure: entry.props.structure, teamMembers: entry.props.teamMembers })`
  11. Return updated entry

**`backend/src/domain/use-cases/schedule-entry/auto-fill-day-gaps.ts`**

- Constructor takes `ScheduleEntryRepository`, `TeamMemberRepository`
- Payload: `{ entryId: string }`
- Response: `{ item: ScheduleEntry, gapReport: DayGapReport }`
- Steps:
  1. Find entry by ID → 404 if not found
  2. Read structure from `entry.props.structure.value`
  3. If structure is empty, return entry as-is with no gaps
  4. Load all team members via `teamMemberRepository.findAll()`
  5. Compute the week range from entry's date using `getMondayOfWeek()` / `getSundayOfWeek()`
  6. Load all week entries via `scheduleEntryRepository.findByDateRange()` for fair distribution counts
  7. Build `assignmentCounts` map from all week entries' team member maps (initialize all team members to 0, then count)
  8. Get currently assigned team member IDs from entry's team member maps
  9. Build lookup maps: `byProfession`, `byProfessionSpecialty` (same as auto-fill)
  10. For each profession requirement in structure:
      - Count already assigned members for this profession
      - **Phase 1 — Fill specialty gaps** (sorted by fewest eligible, most constrained first):
        - For each specialty requirement: count assigned members with this specialty
        - If assigned < required: fill from eligible (matching profession+specialty, not assigned today) sorted by `assignmentCounts` ascending
      - **Phase 2 — Fill remaining general profession gaps**:
        - `remainingToFill = profReq.requiredCount - totalAssignedForProfession`
        - Fill from eligible (matching profession, not assigned today) sorted by `assignmentCounts` ascending
      - Track gaps at both levels
  11. Add newly assigned members to watched list via `entry.props.teamMembers.add()` for each new assignment
  12. Persist via `scheduleEntryRepository.update()`
  13. Return updated entry + gap report

## Shared Validation Logic

The "specialty slot detection" logic is shared between `list-swap-candidates` and `swap-team-member`. Extract it to `backend/src/domain/use-cases/utils/schedule-entry.ts`:

```ts
function determineIsSpecialtySlot(
  structure: StructureData,
  assignedTeamMemberIds: string[],
  allTeamMembers: TeamMember[],
  targetMember: TeamMember,
): boolean
```

- Finds the profession requirement in structure matching `targetMember.profession`
- Finds the specialty requirement matching `targetMember.specialty` (if any)
- Counts assigned members with the same specialty (excluding `targetMember`)
- Returns `true` if `specialtyReq` exists AND `(assignedWithSpecialty - 1) < specialtyReq.requiredCount`

Also extract:

```ts
function validateTeamMemberIsAssignedToEntry(
  entry: ScheduleEntry,
  teamMemberId: string,
): ScheduleEntryTeamMemberMap
```

- Finds the map entity in `entry.props.teamMembers.getItems()` where `teamMemberId` matches
- Throws `ValidationError` with `TEAM_MEMBER_NOT_ASSIGNED_TO_ENTRY` if not found
- Returns the found map entity (useful for the `remove` call in swap)
