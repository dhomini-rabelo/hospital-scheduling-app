# Plan: US-001 — Manage Team Members (Backend)

## Context

The backend has a clean architecture skeleton with base classes (`Entity`, `UseCase`, `DomainError`, `ValidationError`) and Express setup, but no feature implementations yet. This is the first feature: CRUD for hospital team members with profession/specialty validation. The PRD requires team members to have a name, profession, and specialty, with validation that specialties are valid for the chosen profession (AC-001-B).

## Error Handling Approach

Use the **existing error classes** from `backend/src/modules/domain/errors.ts`:
- `ValidationError` — for field-level validation failures (e.g., invalid specialty for a profession)
- `DomainError` with `DangerErrors.NOT_FOUND` — for entity-not-found cases
- **No custom error classes** — all errors use the existing `DomainError`/`ValidationError` system

The error middleware follows the pattern from `plan/docs/error-middleware-sample.md`:
- `ZodError` → 400 with formatted field errors
- `ValidationError` → 400 with `{ [errorField]: [code] }`
- `DomainError` → status from `DangerErrors` map (NOT_FOUND→404, DATA_INTEGRITY→409, etc.) with `{ message: code }`
- Fallback → 500

## API Contract

| Method | Path | Body / Query | Success | Errors |
|--------|------|-------------|---------|--------|
| POST | `/team-members` | `{ items: [{ name, profession, specialty }, ...] }` | 201 + JSON array | 400 invalid |
| GET | `/team-members` | `?profession=DOCTOR` (optional) | 200 + array | -- |
| PUT | `/team-members/:id` | `{ name?, profession?, specialty? }` | 200 + JSON | 404, 400 |
| DELETE | `/team-members/:id` | -- | 204 | 404 |

## Implementation Steps

### Phase 0: Prisma Setup

1. **Install deps** — `npm install prisma --save-dev` and `npm install @prisma/client` in `backend/`
2. **Init Prisma** — `npx prisma init --datasource-provider sqlite`
3. **Define schema** at `backend/prisma/schema.prisma`:
   - `TeamMember` model: `id` (String, uuid), `name`, `profession`, `specialty` (all String), `createdAt`, `updatedAt`
   - `@@map("team_members")`, column names snake_case via `@map`
   - profession/specialty as plain Strings (SQLite has no enums; validation at domain layer)
4. **Update** `backend/.env.development` — add `DATABASE_URL="file:./dev.db"`
5. **Update** `backend/src/infra/services/env.ts` — add `DATABASE_URL: z.string()` to schema
6. **Create** `backend/src/infra/services/prisma.ts` — PrismaClient singleton
7. **Run migration** — `npx prisma migrate dev --name init-team-member`
8. **Add Prisma scripts** to `backend/package.json`
9. **Update** `.gitignore` — add `*.db`, `*.db-journal`

### Phase 1: Domain Layer

10. **Create** `backend/src/domain/entities/team-member.ts`:
    - Enums: `Profession` (DOCTOR, NURSE, TECHNICIAN, SUPPORT_STAFF)
    - Per-profession specialty enums: `DoctorSpecialty`, `NurseSpecialty`, `TechnicianSpecialty`, `SupportStaffSpecialty`
    - `PROFESSION_SPECIALTIES` map (single source of truth for validation)
    - `isValidSpecialtyForProfession()` exported helper
    - `TeamMemberProps`: name, profession, specialty, createdAt, updatedAt
    - `TeamMember` extends `Entity<TeamMemberProps>`:
      - `create(props)` — validates specialty, auto-sets timestamps, throws `ValidationError` if invalid
      - `reference(id, props)` — rebuilds from DB, no validation
      - `update(partial)` — re-validates profession-specialty pair, throws `ValidationError` if invalid

    **Error examples in entity:**
    ```ts
    // Invalid specialty for profession
    throw new ValidationError({
      errorField: 'specialty',
      code: 'INVALID_SPECIALTY_FOR_PROFESSION',
      variables: [props.profession, props.specialty],
    })
    ```

### Phase 2: Adapter Layer

11. **Create** `backend/src/adapters/repositories/team-member-repository.ts`:
    - Abstract class with: `create(teamMember)`, `createMany(items)`, `findById(id)`, `findAll(profession?)`, `update(teamMember)`, `delete(id)`
    - `findById` returns `TeamMember | null`

### Phase 3: Prisma Repository

12. **Create** `backend/src/infra/adapters/repository/prisma-team-member-repository.ts`:
    - Implements `TeamMemberRepository` using prisma singleton
    - Uses `TeamMember.reference()` for reconstruction from DB records
    - `createMany` uses `prisma.teamMember.createManyAndReturn()` for atomic batch creation
    - `findAll` orders by name asc, accepts optional profession filter

### Phase 4: Use Cases

13. **Create** `backend/src/domain/use-cases/team-member/create-team-members.ts`:
    - Accepts `{ items: TeamMemberInput[] }` payload for batch creation
    - Validates specialty for every item before creating any
    - Calls `repo.createMany` to persist all members atomically
    - `ValidationError` propagates naturally if any specialty is invalid

14. **Create** `backend/src/domain/use-cases/team-member/list-team-members.ts`:
    - Calls `repo.findAll` with optional profession filter

15. **Create** `backend/src/domain/use-cases/team-member/update-team-member.ts`:
    - `findById` → if null, throw `DomainError({ errorType: DangerErrors.NOT_FOUND, code: 'TEAM_MEMBER_NOT_FOUND', variables: [id] })`
    - Call `entity.update()` (which re-validates) + `repo.update`

16. **Create** `backend/src/domain/use-cases/team-member/delete-team-member.ts`:
    - `findById` → if null, throw `DomainError` with NOT_FOUND
    - `repo.delete(id)`

### Phase 5: HTTP Layer

17. **Create** `backend/src/infra/http/error-handler.ts`:
    - Express error middleware following the sample from `plan/docs/error-middleware-sample.md`
    - Handles: `ZodError` → 400, `ValidationError` → 400, `DomainError` → status by `errorType` map
    - `DangerErrors` status map: `DATA_INTEGRITY` → 409, `NOT_FOUND` → 404, `FORBIDDEN` → 403, `UNAUTHORIZED` → 401
    - Includes `formatZodError` helper from sample
    - Express 5 auto-forwards async rejections, so routes don't need try/catch

18. **Create** `backend/src/infra/http/presenters/team-member-presenter.ts`:
    - Converts entity to JSON: `{ id, name, profession, specialty, createdAt, updatedAt }`

19. **Create route files** in `backend/src/infra/http/routes/`:
    - `create-team-members.ts` — Zod validates body (`{ items: [...] }` array, min 1), instantiates repo + use case inline (per backend-patterns)
    - `list-team-members.ts` — Zod validates query params
    - `update-team-member.ts` — Zod validates params (uuid) + body (all optional)
    - `delete-team-member.ts` — Zod validates params (uuid)

20. **Update** `backend/src/infra/http/app.ts` — register 4 routes + error handler (after routes)

## Key Design Decisions

- **No custom error classes** — use `ValidationError` and `DomainError` from `@/modules/domain/errors.ts`
- **Profession-specialty validation lives in the entity** (domain invariant), throws `ValidationError`
- **Not-found throws `DomainError`** with `DangerErrors.NOT_FOUND` in use cases
- **Specialty stored as String in Prisma** — SQLite has no enum support; cast on read is safe since domain validates on write
- **Route handlers instantiate use cases inline** — follows backend-patterns.md
- **Express 5 async error handling** — no try/catch in routes; error middleware catches all
- **`update()` re-validates the profession-specialty pair** — if profession changes, current specialty must still be valid
