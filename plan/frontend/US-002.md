# Plan: US-002 Frontend — Define Schedule Requirements

## Context

US-001 (Team Members) is fully implemented on both backend and frontend. US-002 backend is complete with 6 API endpoints for schedule requirement CRUD + enable/disable. The frontend currently has a placeholder page at `/schedule-requirements`. This plan implements the full Schedule Requirements page following the patterns established by US-001.

Schedule requirements are staffing rules per "date reference" (user-defined label like "weekday", "weekend"). Each requirement contains an array of profession requirements, each with a required count and optional specialty sub-requirements. This creates a 3-level nested data structure that makes the form the most complex part of this feature.

## Zod Schemas

```ts
const specialtyRequirementSchema = z.object({
  specialty: z.string().min(1, 'Select a specialty'),
  requiredCount: z.coerce.number().int().positive('Must be at least 1'),
})

const professionRequirementSchema = z.object({
  profession: z.nativeEnum(Profession, { message: 'Select a profession' }),
  requiredCount: z.coerce.number().int().positive('Must be at least 1'),
  specialtyRequirements: z.array(specialtyRequirementSchema).default([]),
}).refine(
  (data) => {
    const specialtySum = data.specialtyRequirements.reduce((acc, s) => acc + s.requiredCount, 0)
    return specialtySum <= data.requiredCount
  },
  { message: 'Specialty counts cannot exceed total required', path: ['specialtyRequirements'] }
)

const createScheduleRequirementSchema = z.object({
  dateReference: z.string().min(1, 'Date reference is required'),
  requirements: z.array(professionRequirementSchema).min(1),
})

const editScheduleRequirementSchema = z.object({
  requirements: z.array(professionRequirementSchema).min(1),
})
```

Key: `z.coerce.number()` handles HTML input string-to-number conversion. `.refine()` on profession schema enforces AC-002-D (specialty sum <= required count). Zod schemas use `z.string()` for specialty (since `Specialty` is a union of multiple enums); submit handlers cast `data.requirements as unknown as CreateScheduleRequirementInput['requirements']` when calling API functions.

## Nested Form Architecture

The 3-level structure is handled by two levels of `useFieldArray`:

1. **Dialog level** (`CreateScheduleRequirementDialog` / `EditScheduleRequirementDialog`):
   - `useForm` + `FormProvider`
   - `useFieldArray({ name: 'requirements' })` → profession rows

2. **`ProfessionRequirementRow`** (uses `useFormContext`):
   - `useFieldArray({ name: \`requirements.${index}.specialtyRequirements\` })` → specialty rows
   - `useWatch` on `requirements.${index}.profession` to react to profession changes

3. **`SpecialtyRequirementRow`** (uses `useFormContext`):
   - `register` for specialty select and required count input

Empty row defaults:
```ts
const EMPTY_PROFESSION_ROW = { profession: '' as Profession, requiredCount: '', specialtyRequirements: [] }
const EMPTY_SPECIALTY_ROW = { specialty: '', requiredCount: '' }
```
