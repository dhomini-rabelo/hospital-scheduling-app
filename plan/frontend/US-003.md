# Plan: US-003 Frontend — View & Create Schedules

## Context

US-001 and US-002 are fully implemented (backend + frontend). US-003 backend is complete with 4 API endpoints: `PUT /schedule-entries`, `GET /schedule-entries`, `DELETE /schedule-entries/:id`, `GET /schedule-overview`. The frontend currently has a placeholder page at `/` (Schedule page). This plan implements the full Schedule page with mini calendar navigation, daily schedule overview, and a "Set Schedule" dialog.

The Schedule page is structurally different from TeamMembers/ScheduleRequirements — those are CRUD list pages. This page is a **dashboard** with a date-navigable overview and a set schedule dialog.

### Date Utilities

**`frontend/src/pages/Schedule/utils/schedule-dates.ts`** — Pure functions for date math (all UTC-based, matching backend `domain/utils/schedule.ts`):
- `formatDateToISO(date: Date): string` — YYYY-MM-DD format using UTC methods
- `getMondayOfWeek(date: Date): Date` — Monday of the week containing date
- `getSundayOfWeek(date: Date): Date` — Sunday of the week containing date
- `getViewRange(): { start: string; end: string }` — Current week Monday → next week Sunday (mirrors backend `getAllowedViewRange`)
- `getSetRange(): { start: string; end: string }` — Today → next week Sunday (mirrors backend `getAllowedSetRange`)
- `getTodayISO(): string` — Today as YYYY-MM-DD
- `generateWeekDays(weekStartMonday: Date): CalendarDay[]` — 7 CalendarDay objects for Mon-Sun
- `generateTwoWeeks(): { currentWeek: CalendarDay[]; nextWeek: CalendarDay[] }` — Both weeks for mini calendar and date selector
- `generateDateRange(startDate: string, endDate: string): string[]` — All dates between start and end inclusive
- `formatDateDisplay(dateStr: string): string` — Human-readable format (e.g., "Fri, Apr 4")

Type:
```ts
export interface CalendarDay {
  date: string           // YYYY-MM-DD
  dayOfMonth: number     // 1-31
  dayOfWeekShort: string // "Mon", "Tue", etc.
  monthShort: string     // "Apr", "May", etc.
  isToday: boolean
  isPast: boolean        // before today (can view but not set)
  isSettable: boolean    // within set range (today → next week Sunday)
}
```

## State Management

**Page level** (`page.tsx`):
- `selectedDate: string` via `useState` (default: `getTodayISO()`)
- `useDialogs<'create'>()` for dialog visibility
- `useQueryClient()` for `invalidateQueries` on mutation success

**ScheduleOverview**: `useAPIRequest<ScheduleOverview>` with `params: { date: selectedDate }` — React Query auto-refetches when date changes (query key includes params).

**SetScheduleDialog**: `useForm` + `FormProvider` + `zodResolver`. Two-stage state (`'structure' | 'team-members'`). Stage 1 validates dates + structure before advancing. Stage 2 shows TeamMemberSelector. Submit only on stage 2.

**Query invalidation**: After `setScheduleEntries` succeeds, invalidate `[API_ROUTES.scheduleEntries.overview]` broadly (prefix matching) so the overview refreshes for any cached date.

## Design Decisions

1. **Full-width mini calendar cells** — `flex-1` distributes day cells evenly across the container, no fixed min-width
2. **Arrow-button week navigation** — simpler, more accessible, clear affordance for "two pages of dates"
3. **"Set Schedule" naming** — reflects the upsert nature of `PUT /schedule-entries` (creates or updates)
4. **Visual date selector grid** — two rows (this week + next week) with toggleable multi-select cells matching the MiniCalendar style, replacing the previous start/end date inputs. Users can select non-contiguous dates (e.g., only weekends)
5. **Two-stage dialog** — Stage 1: dates + requirements + structure. Stage 2: team member assignment. Validates stage 1 before advancing. Keeps the form manageable instead of one long scrollable form
6. **Self-contained page components** — structure rows are duplicated from ScheduleRequirements rather than extracted to shared location, following the codebase convention where pages don't share components
7. **CalendarCog icon** for "Set Schedule" button — visually distinct from the Calendar icon used in the page header
8. **No date library** — codebase has no date-fns/dayjs, and the required functions are minimal pure functions
9. **UTC date math** — matches backend pattern exactly, avoids timezone bugs
